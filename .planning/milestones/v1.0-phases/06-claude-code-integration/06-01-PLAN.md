Include a note that `--format json` is the default and is required for machine-readable output.

**All flags with descriptions** (mirror cli.rs defaults exactly):
- `--limit N` — max results (default: 10)
- `--min-score F` — minimum score threshold 0.0-1.0 (optional)
- `--mode` — `hybrid` (default, fuses BM25+vector via RRF), `semantic` (vector ANN only), `fts` (full-text BM25 only)
- `--path-filter STR` — restrict to files under this path prefix
- `--tag-filter STR` — restrict to files with this frontmatter tag
- `--context N` — include N surrounding chunks before/after each match (default: 0)
- `--data-dir PATH` — override index data directory (env: LOCAL_INDEX_DATA_DIR)

**Example invocation:**
```sh
local-index search "how to configure git rebase" --limit 5 --mode hybrid
```

**Annotated example JSON output** — a JSON array with 2 example result objects. Each
field must have an inline comment explaining what it means:

```json
[
  {
    "chunk_text": "...",         // The actual text content of this chunk
    "file_path": "git/rebase.md", // Path relative to the vault root
    "heading_breadcrumb": "## Git Workflow > ### Rebase Strategy",  // Full heading path
    "similarity_score": 0.87,    // 0.0-1.0; hybrid mode uses RRF score (higher = better match)
    "line_range": { "start": 42, "end": 67 },  // Line numbers in the source file
    "frontmatter": {             // YAML frontmatter parsed from the source file
      "tags": ["git", "workflow"],
      "aliases": [],
      "date": "2024-03-15"
    }
  }
]
```

**Score interpretation guidance:**
- Hybrid RRF scores: 0.01-0.10 range is typical; scores above 0.05 are strong matches
- Semantic cosine scores: 0.0-1.0; above 0.75 is a strong semantic match
- FTS scores: rank-based; higher = more keyword overlap
- Empty array means no results above the threshold (try lowering `--min-score` or broadening the query)

**Common follow-up patterns** (numbered list):
1. Read the source file at the returned line range for more context: use Read tool on `file_path` at lines `line_range.start` through `line_range.end`
2. Use `heading_breadcrumb` to understand the document structure and locate the relevant section
3. Run a follow-up search narrowed with `--path-filter` to find related notes in the same directory
4. Use `--tag-filter` to restrict to notes with a specific topic tag

**Error cases** (how Claude should handle each):
- Exit non-zero + "VOYAGE_API_KEY not set" → tell the user to set the env var and try again
- Exit non-zero + "no index found" or "table not found" → the vault hasn't been indexed yet; use the reindex skill
- Empty JSON array `[]` → no results; suggest broadening the query or removing score threshold
- Exit non-zero + network error → Voyage AI API unreachable; check connectivity

Before invoking `local-index index`, Claude MUST follow this decision tree:
1. Check if `$OBSIDIAN_VAULT` is set in the environment
2. If set, use it: `local-index index "$OBSIDIAN_VAULT"`
3. If `$OBSIDIAN_VAULT` is not set, check `$LOCAL_INDEX_VAULT`
4. If that is set, use it: `local-index index "$LOCAL_INDEX_VAULT"`
5. If neither is set, ask the user: "What path should I reindex?"

Include a note that the binary does NOT auto-discover the vault path — it must be
provided explicitly.

Invocation:
```sh
local-index index <PATH> [--force-reindex] [--data-dir PATH]
```

Flag descriptions:
- `--force-reindex` — skip content hash check; re-embed all chunks even if unchanged
- `--data-dir PATH` — override index data directory (env: LOCAL_INDEX_DATA_DIR)

Example invocations:
```sh
# Using env var (preferred)
local-index index "$OBSIDIAN_VAULT"

# Force reindex (e.g., after changing embedding model)
local-index index "$OBSIDIAN_VAULT" --force-reindex
```

What to expect during indexing (progress output goes to stderr; exit 0 on success).

Error cases:
- `VOYAGE_API_KEY` not set → embedding will fail; set env var
- PATH argument missing → binary exits with usage error; must provide vault path
- Network error during embedding → exponential backoff is automatic; if repeated failures, check API key and connectivity
- "model mismatch" warning → existing index used a different embedding model; rerun with `--force-reindex`

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| skill file → Claude Code executor | Skill files are treated as trusted instructions by Claude Code; malicious content could direct unintended Bash invocations |
| shell scripts → system PATH | Scripts invoke `local-index` by name; PATH manipulation could redirect execution |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-01 | Tampering | scripts/*.sh | accept | Scripts are repo-committed, version-controlled; no privilege escalation possible via pass-through wrappers |
| T-06-02 | Information Disclosure | .claude/skills/search.md | accept | Skill files contain no secrets; VOYAGE_API_KEY is only referenced by name, never embedded |
| T-06-03| Spoofing | PATH-based binary resolution | accept | `local-index` binary from `cargo install` is in user-controlled PATH; same risk as any CLI tool; no mitigation warranted for a local dev tool |
| T-06-04 | Elevation of Privilege | reindex.md path strategy | mitigate | Skill file explicitly instructs Claude to check env vars before asking user; prevents unintended filesystem access by requiring explicit path confirmation when env vars are absent |
</threat_model>
