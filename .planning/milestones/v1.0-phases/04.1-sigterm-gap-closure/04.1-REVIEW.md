# Phase 04.1: Code Review Report

**Reviewed:** 2026-04-10
**Depth:** standard
**Files Reviewed:** 2
**Status:** issues_found

## Summary

This phase adds SIGTERM handling to the shutdown module and accompanying integration tests. The implementation is correct and clean: `setup_shutdown` registers both SIGINT and SIGTERM before entering `select!`, which ensures signals arriving before the await point are queued by the OS rather than dropped. The `CancellationToken` pattern integrates correctly with the rest of the daemon (verified via `src/daemon/mod.rs`). The `tokio-util` `rt` feature correctly enables `tokio/sync` transitively, so `CancellationToken` compiles cleanly.

One warning-level issue exists: the two new tests are susceptible to a race condition when run in parallel, which is the default for `cargo test`. This will cause intermittent flaky failures in CI.

Two info-level items cover a force-exit code convention and a redundant dev-dependency.

## Warnings

### WR-01: Parallel test race on OS signal handlers

**File:** `src/daemon/shutdown.rs:41-70`

**Issue:** Both `test_sigterm_cancels_token` (line 41) and `test_sigint_still_cancels_token` (line 57) call `setup_shutdown()`, which spawns a background task that installs an OS-level signal handler covering the entire process. By default `cargo test` runs tests in parallel across threads. When both tests execute concurrently:

- Two SIGTERM handler tasks are registered simultaneously.
- When one test sends `libc::kill(getpid(), SIGTERM)`, either handler task (from either test) may consume the signal first — the winner is non-deterministic.
- The test whose token gets cancelled is effectively random. The other test's `tokio::time::timeout` expires after 500 ms and panics.

This produces flaky CI failures that are hard to diagnose because they are load-dependent.

**Fix:** Add `serial_test` to `[dev-dependencies]` and mark both signal tests `#[serial]` so they are forced to run sequentially:

```toml
# Cargo.toml [dev-dependencies]
serial_test = "3"
```

```rust
use serial_test::serial;

#[tokio::test]
#[serial]
async fn test_sigterm_cancels_token() { ... }

#[tokio::test]
#[serial]
async fn test_sigint_still_cancels_token() { ... }
```

Alternatively, if the project prefers not to add a dependency, use `#[ignore]` and document that these tests must be run with `cargo test -- --test-threads=1 --ignored`.

## Info

### IN-01: Force-exit uses code 1 instead of conventional 130

**File:** `src/daemon/shutdown.rs:30`

**Issue:** `std::process::exit(1)` is called when a second SIGINT arrives (force-exit path). The POSIX convention for a process that exits due to SIGINT is exit code `130` (128 + signal number 2). Using `1` makes the forced exit indistinguishable from an application error in shell scripts that check `$?`. This has no runtime correctness impact but can confuse operators and automation.

**Fix:**
```rust
std::process::exit(130);
```

### IN-02: `serde_json` duplicated in `[dev-dependencies]`

**File:** `Cargo.toml:83`

**Issue:** `serde_json = "1.0"` appears in both `[dependencies]` (line 29) and `[dev-dependencies]` (line 83). Cargo deduplicates these automatically, so there is no build or runtime impact. However, the redundant entry can mislead readers into thinking `serde_json` is only needed for tests, and it creates a second version constraint to keep in sync during upgrades.

**Fix:** Remove the `serde_json` line from `[dev-dependencies]`. The `[dependencies]` entry already makes it available in test code.
